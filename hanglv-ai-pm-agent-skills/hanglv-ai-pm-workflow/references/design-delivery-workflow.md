# 设计到交付自动化工作流

用于把航旅纵横页面需求变成高保真 HTML 原型、原型截图、PRD 内原型链接、Figma 设计稿、UI 审核稿和前端 PR。

## 核心原则

1. Figma 是设计评审源，HTML 是可交互原型源，前端代码是生产实现源。
2. 不要承诺 Figma、HTML、前端代码可以无损双向同步；同步必须依赖 design token、组件映射和差异审核。
3. PRD 中的原型交付不能只有链接，必须同时放可预览截图，便于文档离线评审和快速扫读。
4. 所有自动修改都需要人工 gate：
   - PM 确认方案/HTML 原型。
   - UI 确认 Figma 设计稿。
   - 前端确认 GitHub PR。
5. 对航旅纵横移动端页面，必须保持交易效率感和既有设计语言，不做泛 SaaS、营销页或玻璃拟态。
6. 是否生成 HTML 原型要先判断：需要原型才生成；不需要原型时不要为了形式生成 HTML。但只要讨论已有 Figma 页面，就应按需读取页面现状。
7. 原型分两类处理：
   - 存量页面改造：先还原、后修改，尽量复刻原页面规范，再在还原稿上做需求改动。
   - 0 到 1 新页面：不需要还原旧图，但必须遵循 `hanglv-c-end-ui-skill` / `hanglvzongheng-design-spec` 的设计规范和页面模式。

## 触发判断

必须生成 HTML 原型的情况：

- 用户明确说要原型图、HTML 原型、交互 Demo、页面改版、完整 PRD、给 UI/研发评审材料。
- 需求涉及新增页面、重排页面结构、关键交互变化、弹层/抽屉/状态切换。
- PRD 中有“交互原型图/UE”且不是纯后台规则或纯策略分析。

不需要生成 HTML 原型的情况：

- 用户只要问题诊断、方案定义卡、客诉聚类、指标分析、策略建议、面试口径。
- 只讨论业务规则，不涉及页面结构或交互变化。

不生成 HTML 原型时：

- 不调用 Figma MCP 去生成 HTML。
- 若用户给了 Figma 链接或问题依赖现有页面，仍应使用 Figma MCP 读取 screenshot、metadata 或 design context，提取 CurrentPageSpec 后再输出。

## 输入要求

至少需要：

- 已确认的问题卡片、方案定义卡或 PRD。
- 目标页面所属链路：搜索、列表、舱位、乘机人、订单确认、支付、出票、退改、报销等。
- 原型类型判断：存量页面改造 / 0 到 1 新页面。

更好的输入：

- 存量页面改造：目标页面的 Figma 链接或 node-id、当前截图、线上页面。
- 0 到 1 新页面：页面类型、目标用户、主任务、关键状态、参考页面模式。
- 页面当前截图或 Figma 节点。
- 页面任务、关键指标、不可破坏的主链路。
- 现有前端仓库路径、页面路由、组件路径。
- design token 文件或 Figma variables。
- UI 修改后的 Figma 链接。

## 阶段 0：判断原型类型

先判断任务属于哪一类：

| 类型 | 判断条件 | 处理方式 |
| --- | --- | --- |
| 存量页面改造 | 用户给 Figma 链接、截图、线上页面，或说“在原有基础上改” | 用 Figma MCP 读取现状，先还原再修改 |
| 0 到 1 新页面 | 无现有页面，或用户说“新做一个页面/从 0 到 1” | 不还原旧图，按航旅纵横设计规范和页面模式生成 |

如果判断不清，优先按“是否存在可读取的现有页面”处理；有现有页面就先读现状，没有现有页面就按 0 到 1 新页面处理。

## 阶段 1A：存量页面改造，用 Figma MCP 获取页面现状

当需求基于现有页面或用户给出 Figma 链接时，必须先读取真实设计上下文，不要凭空画。这个阶段既适用于需要生成 HTML 原型的任务，也适用于只需要诊断/方案但必须了解页面现状的任务。

执行：

1. 用 Figma MCP 获取目标 node 的 design context、metadata、screenshot。
2. 提取 CurrentPageSpec：
   - 页面类型。
   - 页面区块。
   - 关键组件。
   - 交互状态。
   - 当前主链路。
   - 可改区域和不可改区域。
   - 原页面视觉规范：颜色、字号、间距、圆角、按钮样式、卡片结构、图标风格。
3. 若页面是机票结果页或舱位页，必须保留航旅特定结构：
   - 机票结果页：蓝色行程上下文、日期条、筛选条、航班卡。
   - 舱位页：航班摘要、舱位卡、价格、退改签、订购按钮。

## 阶段 1B：0 到 1 新页面，读取航旅设计规范

当没有可还原的现有页面时，不调用 Figma MCP 去“生成 HTML”。此时必须读取并应用：

- `hanglv-c-end-ui-skill`：整页生成时至少读取 `references/design-rules.md`、`references/page-patterns.md`、`references/components.md`。
- `hanglvzongheng-design-spec`：至少读取 `references/design-spec.md`；涉及具体页面组合时读取 `references/screen-patterns.md`。

执行：

1. 判定页面类型：列表/结果页、舱位选择页、订单详情页、表单页、弹层页、首页/运营页等。
2. 选择最接近的航旅页面模式和组件组合。
3. 明确状态覆盖：default、selected、disabled、loading/empty/error、弹层/抽屉/确认态。
4. 明确不可偏离规则：移动端交易效率、浅色背景、白色内容面、品牌绿主动作、机票链路受控蓝色上下文。

## 阶段 1C：存量页面先还原现状，再定义改动

生成原型前必须先形成两个中间判断：

1. CurrentPageSpec：现页面是什么样。
   - 页面骨架、组件层级、文案密度、按钮语义、颜色和卡片结构。
   - 哪些区域不可破坏，例如航班摘要、舱位卡主结构、价格与订购按钮。
2. PageChangeSpec：本次只改什么。
   - 新增、删除、移动、重命名、合并的模块。
   - 哪些交互状态新增。
   - 哪些原有组件继续复用。
   - 哪些视觉规则必须保持。

执行原则：

- HTML 原型先还原原 Figma 页面基线，再叠加方案改动。
- 优先沿用原页面的布局、字号、颜色、圆角、卡片结构和按钮语义。
- 不因为“更现代”而替换成新的设计语言。
- 只有当原页面结构无法支撑需求时，才允许做结构性重排，并在 PRD 中说明原因、收益和风险。

## 阶段 2：生成 HTML 高保真原型

HTML 原型用于 PM/业务/UI 前置评审，要求可交互、可运行、可批改。

输出要求：

1. 生成独立 HTML 文件，优先放到 `/Users/weizheng/Downloads/`。
   - 存量页面改造：HTML 的第一目标是“基于原稿还原后的改版”，不是重新设计；新增模块应嵌入原页面结构，尽量使用原有组件语言。
   - 0 到 1 新页面：HTML 的第一目标是“符合航旅纵横设计规范的可交互新页面”，不是通用 Web/SaaS 风格页面。
2. 覆盖关键状态：
   - default。
   - loading/empty/error，如相关。
   - selected/disabled。
   - 弹窗、抽屉、确认态。
   - 业务异常态，如价格变化、审核失败、超时、退款中等。
3. 不只是静态还原，必须能点击演示关键路径。
4. 使用航旅设计规则和 design token 占位：
   - 颜色、字号、间距、圆角、按钮语义尽量 token 化。
   - 缺 token 时标注待确认，不要乱造规范。
5. 原型生成后，必须用浏览器或截图工具截取至少一张 PNG：
   - 默认态截图：`<prototype-name>-default.png`。
   - 关键交互截图：如 `<prototype-name>-compare-sheet.png`、`<prototype-name>-expanded.png`，按需生成。
   - 截图优先与 HTML 放在同一目录，默认 `/Users/weizheng/Downloads/`。
6. 同步生成或更新 PRD/方案文档时，必须把 HTML 原型链接写进文档的“交互原型图/UE/链接”位置，并紧跟 Markdown 图片嵌入截图：
   - `原型链接：[点击打开 HTML 原型](/abs/path/prototype.html)`
   - `![原型默认态](/abs/path/prototype-default.png)`

PM 确认前，不写入 Figma。

## 阶段 3：确认后写入 Figma

用户确认 HTML 原型后，才使用 Figma MCP 写入 Figma。

写入规则：

1. 优先在目标 Figma 文件中新建 page/section，避免覆盖 UI 原稿。
2. 命名格式：
   - `AI 原型 - <需求名> - <日期>`
   - frame 名称包含页面和状态。
3. 写入内容应包括：
   - 主页面。
   - 关键状态页。
   - 弹窗/抽屉状态。
   - 必要的交互说明。
4. 交付给用户 Figma 链接，让用户转给 UI 审核。

边界：

- Figma MCP 写入的是可审核设计草稿，不等同于 UI 最终稿。
- UI 可以在 Figma 中修改视觉、组件、间距和状态。

## 阶段 4：UI 修改后同步回 HTML

UI 审核并修改 Figma 后，需要把修改同步回 HTML。

同步方式：

1. 再次用 Figma MCP 读取 UI 修改后的 node。
2. 对比旧 HTML 原型和新 Figma：
   - 视觉 token 差异：颜色、字号、间距、圆角。
   - 组件结构差异：新增/删除模块、顺序变化。
   - 文案差异。
   - 状态差异。
3. 能 token 化的改动，更新 HTML 的 CSS variables/design token。
4. 组件结构变化，更新 HTML DOM 和交互逻辑。
5. 输出同步报告：
   - 已同步。
   - 需人工确认。
   - 无法自动同步。

不要承诺“Figma 一改 HTML 自动 100% 无损变化”。更可靠的说法是：基于 Figma MCP + design token + 组件映射做自动差异同步，并保留人工确认。

## 阶段 5：HTML 转前端技术栈

HTML 确认后，再转为航旅纵横前端实现。

执行前必须确认：

- 前端仓库路径。
- 页面路由。
- 技术栈和框架。
- 组件库和 design token 使用方式。
- 是否需要 Weex/Vue/React/小程序/H5 等具体实现。

转换规则：

1. HTML 只作为原型参考，不直接当生产代码。
2. 优先复用现有组件和 token。
3. 保持业务逻辑与 PRD 一致。
4. 新增/修改埋点按 PRD 的入口定义表格落代码。
5. 本地跑 lint/test/build 或最小可行验证。
6. 创建 GitHub PR，PR 描述包含：
   - 需求背景。
   - 改动范围。
   - 页面截图/原型链接/Figma 链接。
   - 测试结果。
   - 风险点。

## 推荐自动化架构

```text
PRD/方案卡
  -> Figma MCP 读取现状
  -> CurrentPageSpec
  -> HTML 高保真原型
  -> PM 确认
  -> Figma MCP 写入设计草稿
  -> UI 审核修改
  -> Figma MCP 读取修改稿
  -> design token / 组件差异同步
  -> HTML 更新
  -> 转前端技术栈
  -> GitHub PR
  -> 前端 review
```

## 风险

1. Figma 设计稿、HTML 原型、前端代码容易漂移，需要明确版本和同步记录。
2. 如果没有组件映射，AI 容易生成一次性 HTML，无法稳定转前端代码。
3. 如果 design token 不完整，视觉同步只能部分自动化。
4. 如果 UI 在 Figma 中做了结构性重设计，HTML 需要重新生成或半自动同步，不能只改 token。
5. 直接自动推代码风险高，必须走 GitHub PR 和前端 review。
