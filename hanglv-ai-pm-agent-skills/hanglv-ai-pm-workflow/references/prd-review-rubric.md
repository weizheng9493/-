# PRD 评审标准

## 好 PRD 的标准

1. 问题明确，并且真的是一个问题。
2. 有竞品分析、用户调研、客诉或数据作为依据。
3. 前置思考清楚：解决问题后的收益大概是什么。
4. 方案设计完整，逻辑清晰。
5. 业务逻辑和交互逻辑都清楚。
6. 边界场景、异常状态、兜底处理考虑全面。
7. 符合产品设计原则，体验自然简洁。
8. 能降低用户理解成本和操作成本。
9. 从转化角度看，尽量降低下单阻塞，能刺激用户下单欲望。
10. 有埋点、效果评估和后续复盘方式。

## 差 PRD 的典型问题

1. 背景空洞，只写“为了提升体验”。
2. 没有收益测算，也没有收益测算口径。
3. 用户场景和需求描述不准确。
4. 把用户吐槽直接等同于需求，没有判断本质问题。
5. 交互逻辑只写默认流程，不写异常/边界。
6. 业务规则不清，研发和设计无法执行。
7. 没有不做范围，方案容易膨胀。
8. 没有埋点，无法判断上线后是否有效。

## 评分表

| 维度 | 0 分 | 1 分 | 2 分 |
| --- | --- | --- | --- |
| 问题清晰度 | 问题空泛 | 有问题但不够聚焦 | 问题明确且证据充分 |
| 用户场景 | 未说明 | 有场景但不具体 | 用户、任务、阻碍清楚 |
| 数据/调研 | 无 | 有零散证据 | 多源证据或量化支撑 |
| 收益 | 无 | 有方向无口径 | 有指标和测算口径 |
| 方案完整度 | 功能堆砌 | 主流程清楚 | 主流程、异常、边界完整 |
| 业务逻辑 | 不清楚 | 部分清楚 | 规则、依赖、兜底清楚 |
| 交互逻辑 | 只写页面 | 有流程 | 状态、反馈、跳转清楚 |
| 体验原则 | 可能增加负担 | 基本合理 | 自然简洁、低成本、少阻塞 |
| 埋点复盘 | 无 | 有指标 | 指标、口径、触发时机清楚 |

总分建议：

- 15-18：可进入评审。
- 10-14：需要补证据或补逻辑。
- 0-9：不建议进入评审，应退回重写。
