# 问题雷达规则

## 聚类原则

按“用户任务 + 阻碍 + 场景”聚类，不按关键词机械合并。

示例：

- “票价更新需要重新搜索导致下不了单”应聚到：价格刷新/交易阻塞/购票链路稳定性。
- “退票扣费不清楚”“改签费和退款金额不一致”应聚到：退改价格解释不清。
- “开发票麻烦”“行程单迟迟开不出来”应聚到：报销凭证流程与状态透明度不足。

## 问题卡片模板

```md
## 问题标题

### 问题一句话
用户在什么场景下，被什么阻碍影响了什么任务。

### 用户原声
- 原声 1
- 原声 2

### 发生场景
页面/流程/用户任务。

### 证据
- 标签频次：
- 低分样本：
- 客诉案例：
- 数据异常：

### 初步根因
区分已确认、推测、待验证。

### 影响判断
用户影响、业务影响、交易阻塞、客服成本。

### 优先级
P0/P1/P2/P3。

### 置信度
高/中/低，并说明依据。

### 下一步
进入方案生成 / 补数据验证 / 进入观察池。
```

## 优先级口径

- P0：阻断支付、出票、退改、报销等核心任务，或有合规/舆情风险。
- P1：显著影响转化、信任、售后成本，高频且有明确场景。
- P2：影响体验但不明显阻断核心任务，或证据还不充分。
- P3：低频、边缘、纯情绪或暂无法产品化处理。

## 证据强度

- 高：多源一致，有量化数据、原声、页面/流程定位。
- 中：有原声和标签频次，但缺核心业务数据或根因验证。
- 低：只有零散反馈、单一案例或推测。
