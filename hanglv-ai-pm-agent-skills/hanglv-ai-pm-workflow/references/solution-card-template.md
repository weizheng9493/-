# 方案定义卡模板

PRD 之前必须先输出方案定义卡，避免直接生成大而空的 PRD。

```md
# 方案定义卡：<方案名称>

## 1. 问题
- 用户问题：
- 业务问题：
- 证据：
- 置信度：

## 2. 目标
- 用户价值：
- 产品/业务价值：
- 本期目标：

## 3. 范围
- 涉及页面/流程：
- 涉及用户：
- 不做范围：

## 4. 核心方案
1. 改动一：
2. 改动二：
3. 改动三：

## 5. 业务逻辑
- 触发条件：
- 判断规则：
- 兜底规则：
- 依赖系统：

## 6. 交互逻辑
- 默认状态：
- 异常状态：
- 空状态：
- 错误/失败：
- 用户反馈：

## 7. 收益与指标
- 北极星/主指标：
- 过程指标：
- 反向指标：
- 收益测算口径：

## 8. 风险与劣势
- 用户风险：
- 业务风险：
- 技术/依赖风险：
- 可能带来的副作用：

## 9. 验证计划
- 上线前验证：
- A/B 或灰度：
- 上线后复盘：
```

## 方案判断标准

好方案应该：

1. 针对真实问题，不是为了做功能而做功能。
2. 不增加主链路不必要阻塞。
3. 用户理解成本和操作成本更低。
4. 业务规则和交互逻辑清晰。
5. 有明确收益或验证口径。
6. 明确不做什么。

差方案常见问题：

1. 只写“优化体验”，没有具体问题。
2. 承诺业务做不到的能力。
3. 增加解释但破坏交易效率。
4. 只考虑默认状态，不考虑异常/失败/边界。
5. 没有指标和复盘方式。
