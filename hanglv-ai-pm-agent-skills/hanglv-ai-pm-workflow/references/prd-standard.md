来源：用户提供的《魏政的产品文档需求规范》与后续人工评审规则。

## PRD 固定结构

```md

## 0. 文档变更日志 & 需求排期 & 链接
- 时间：
- 版本号：
- 变更人：
- 主要变更：
- 负责人/排期/备注：

## Part 1：为什么要做这个需求

### 1. 背景
从现状、用户需求场景、用户反馈、数据分析、历史尝试、竞品分析说明为什么要做，是否应该现在做。

### 1.1 要解决的问题
简单说明需求要解决的问题，让读者快速抓住重点。

### 1.2 用户调研/客诉结论
说明样本来源、典型原声、关键结论。

### 1.3 数据分析结论
列关键数据，不展开完整分析过程。

### 1.4 竞品分析结论
列竞品能力和关键结论，推演背后动机，再与自身问题对比。

### 2. 需求目标/预期收益
目标必须可衡量，包含用户价值和产品价值。

#### 2.1 需求目标

#### 2.2 收益预估

#### 2.3 收益衡量标准/效果评估

## Part 2：需求要做成什么样

### 3. 需求详情

#### 3.1 流程图/泳道图
复杂流程才写，简单改动可以用文字流程。

#### 3.2 交互原型图
先判断本需求是否需要原型图：
- 需要原型图：页面改版、新页面、交互方案、完整 PRD、给 UI/研发评审、用户明确要求原型图或 HTML 原型。必须生成可打开的 HTML 原型，并在此处放原型链接和原型截图。
- 不需要原型图：问题诊断、方案定义卡、策略建议、客诉聚类、指标分析。可以不生成原型，也不要为了形式虚构链接。
- 存量页面改造：如果基于现有 Figma 页面改版，即使不生成原型，也应按需用 Figma MCP 读取页面现状后再写 PRD；如果生成原型，必须先还原原页面，再在还原稿上做改动。
- 0 到 1 新页面：没有现有页面可还原时，不需要先还原旧图；必须遵循航旅纵横设计规范 skill，并在 PRD 中说明采用的页面模式和组件规则。

需要原型图时，本小节推荐格式：

```md
原型链接：[点击打开 HTML 原型](/abs/path/prototype.html)

默认态截图：

![原型默认态](/abs/path/prototype-default.png)

关键交互状态截图：

![权益对比弹层](/abs/path/prototype-compare-sheet.png)
```

截图要求：
- 至少包含一张默认态 PNG 截图。
- 如果需求包含弹层、抽屉、展开/收起、确认态等关键交互，按需补充关键状态截图。
- 截图文件优先和 HTML 原型放在同一交付目录，常用 `/Users/weizheng/Downloads/`。

原型说明要求：
- 存量页面改造：先写“保留原页面规范”，例如顶部航班摘要、舱位卡结构、价格与订购按钮、蓝色机票上下文层；再写“本次改动区域”，例如新增模块、调整文案、移动权益、增加弹层。
- 0 到 1 新页面：先写“采用的航旅页面模式/组件规则”，例如表单页、舱位页、详情页、Bottom Sheet；再写“核心页面结构和状态覆盖”。
- 不要把存量页面改造成脱离现有 Figma 页面的一套新 UI，也不要把 0 到 1 新页面做成泛 SaaS 或营销站风格。

#### 3.3 功能点详述
按下方表格写。需要原型时，UE 栏必须放 HTML 原型链接、截图链接或对应状态说明；不需要原型时，UE 可写“无需原型/待 UI 评审”，但需求详述必须清楚。

| 功能点 | UE | 需求详述 |
| --- | --- | --- |
| 1. <功能点名称> | HTML 原型链接 + 截图/无需原型 | 1. 规则一<br>2. 规则二<br>&nbsp;&nbsp;a. 子规则<br>&nbsp;&nbsp;b. 子规则 |

#### 3.4 UI 稿
可以先空着，用“待补充”占位。

## 写作要求

1. 先讲问题和价值，再讲方案。
2. 背景不能空洞，必须有用户反馈、数据、竞品或业务事实。
3. 收益测算没有绝对数字时，也要给计算口径和所需数据。
4. 功能说明必须包含业务逻辑和交互逻辑，但表达要尽量简洁易懂，避免写成大段散文。
5. “需求详情”优先用 `功能点 / UE / 需求详述` 表格：左侧拆功能点，中间放原型/截图占位，右侧按 1、2、a、b 写规则。
6. 一个功能点不要塞太多主题；如果规则超过 5 条，拆成多个功能点。
7. 复杂流程要写清状态、边界、异常和兜底；但不要为了全面堆无关边缘场景。
8. 原型图按需生成：需要原型时必须先生成 HTML 原型、截图并写入链接；不需要原型时可以不写，但不要虚构截图或链接。
9. 埋点优先用“入口定义”表格，字段包括入口类型、事件名称、页面、模块、服务卡片、点击区域、触发口径。
10. 埋点要服务于上线后效果评估，不为埋点而埋点。
11. 需求必须说明不做范围，防止范围膨胀。

## 需求详述写法示例

```md
| 功能点 | UE | 需求详述 |
| --- | --- | --- |
| 1. 审核不通过结果展示 | 待补充 | 1. 当病退材料审核不通过时，展示审核结果和不通过原因。<br>2. 若缺少材料，逐项展示缺失材料：<br>&nbsp;&nbsp;a. 诊断证明。<br>&nbsp;&nbsp;b. 就诊缴费发票。<br>&nbsp;&nbsp;c. 身份证与材料同框照片。<br>3. 展示补材料截止时间，格式：请于 YYYY-MM-DD HH:mm 前补充材料。 |
| 2. 补材料入口 | 待补充 | 1. 未超过补材料截止时间时，展示“补充材料”主按钮。<br>2. 超过截止时间后，按钮置灰，不直接消失。<br>3. 点击置灰按钮时，说明无法提交的原因，并提供客服入口。 |
```
