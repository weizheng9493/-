# 问题诊断框架

## 诊断问题

每个问题进入方案前，必须回答：

1. 这是不是一个真实产品问题，还是用户对规则/价格/航司能力的误解？
2. 问题发生在用户旅程的哪一段：搜索、列表、舱位、乘机人、订单确认、支付、出票、退改、报销、客服？
3. 用户原本想完成什么任务？
4. 阻碍来自哪里：信息不清、流程复杂、规则限制、系统 bug、供给问题、航司协同、客服履约？
5. 产品能解决到什么程度？
6. 哪些方向不能做或需要谨慎？

## 类型判断

| 类型 | 典型表现 | 处理方向 |
| --- | --- | --- |
| 信息解释问题 | 票价、退改、报销、状态不清楚 | 强化解释、状态、原因、规则入口 |
| 流程效率问题 | 路径长、步骤多、反复跳转 | 缩短路径、默认值、渐进式填写 |
| 交易阻塞问题 | 支付、出票、价格刷新、库存导致无法下单 | 兜底、重试、替代方案、明确反馈 |
| 业务规则问题 | 航司规则、退改限制、发票主体限制 | 规则透明化，不虚假承诺 |
| 技术稳定性问题 | Bug、卡顿、接口失败、状态不同步 | 修复、监控、错误提示、降级 |
| 客服履约问题 | 电话不通、处理慢、结果不通知 | 自助查询、状态通知、人工升级 |
| 供给/航司协同问题 | 航司拒开、航变、行李规则差异 | 责任边界、凭证、引导、协同机制 |

## 诊断输出

是/否，以及理由。
