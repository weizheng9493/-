# 工作流治理规则

## 第一性原则

本仓库的核心不是存文件，而是让团队可以围绕同一套 AI PM 工作流持续改进。任何提交都应该满足至少一个目标：

- 提高工作流复用性。
- 降低团队协作成本。
- 提高智能体输出质量。
- 沉淀可追踪的案例和经验。
- 改善发布和审核质量。

## 变更类型

| 类型 | 示例 | 审核重点 |
| --- | --- | --- |
| 文档更新 | 修改 `docs/` | 表述是否清晰、是否影响团队使用 |
| 工作流更新 | 修改 `workflows/` | 步骤是否可执行、边界是否清楚 |
| Prompt 更新 | 修改 `prompts/` | 输入输出是否明确、是否可复用 |
| Skill 更新 | 修改 `hanglv-ai-pm-agent-skills/` | 是否破坏已有引用和行为 |
| 产物提交 | 新增 `artifacts/` | 来源、风险、人工校验是否清楚 |
| 发布准备 | 修改 `CHANGELOG.md` | 版本范围是否准确 |

## PR 审核要求

每个 PR 至少说明：

- 本次更新点。
- 更新原因。
- 影响范围。
- 验证方式。
- 风险和待确认点。

## 正式内容和待沉淀内容

| 状态 | 存放位置 | 含义 |
| --- | --- | --- |
| 正式内容 | `docs/`、`workflows/`、`prompts/`、`hanglv-ai-pm-agent-skills/` | 团队可直接使用 |
| 待沉淀内容 | `artifacts/` | 需要进一步审核、整理或产品化 |
| 示例内容 | `examples/` | 用于说明规范和最佳实践 |

## 质量门槛

- 不能提交密钥、账号、Cookie、未脱敏客户信息。
- 不确定结论必须标注待确认。
- AI 生成内容不能直接升格为正式规范。
- 对团队流程有影响的修改，需要说明迁移成本。
- 代码或脚本必须说明验证方式。
